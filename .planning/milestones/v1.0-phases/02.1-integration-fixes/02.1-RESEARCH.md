## Summary

Phase 2.1 is a surgical gap-closure phase. The milestone audit identified three concrete integration defects in the Phase 1/2 completed code: a wrong default species string in `plant_data.py`, a standalone hardcoded species dict in `parity.py` that duplicates the YAML registry without being backed by it, and a stale `ct` brand reference in `downloader.py`. All three bugs are fully diagnosed — the root causes are documented in the audit, the fix sites are pinpointed to exact file:line, and the fix strategies are unambiguous.

There is no research uncertainty here. The codebase has been read and the fix is a pure code-change exercise: one default string fix, one module refactor to call `_species.py` instead of a local dict, and one string literal update. No new libraries are needed. No new patterns are needed. The existing `_species.py`/`species_registry.yaml` infrastructure is the correct destination for species resolution and already works correctly for all other call sites.

The only design decision required is how `_normalize_mygene_species()` in `parity.py` should call into `_species.py`. The answer is: use `resolve_species_taxon()` to get the numeric taxon ID, then format it as a string. The existing decision that plant species use numeric taxon ID strings with MyGene.info (recorded in STATE.md) remains correct.

**Primary recommendation:** Three targeted edits; run existing tests after each; no new tests required beyond asserting the fixed behaviour.

## Architecture Patterns

### Pattern 1: Species Resolution via `_species.py`

**What:** All species string normalization goes through `ct.tools._species` which reads `species_registry.yaml`. Tools never maintain their own species lookup tables.

**When to use:** Any time a tool needs to convert a species name/abbreviation/common-name to a taxon ID or canonical binomial.

**Current state (broken):** `parity.py` defines `_MYGENE_SPECIES_MAP` (lines 41-91) — a standalone dict with 30+ entries that duplicates registry content. New species added to YAML will not appear in MyGene lookups.

**Fixed state:** `_normalize_mygene_species()` calls `resolve_species_taxon()` from `_species.py`. If the species resolves to a nonzero taxon ID, return it as a string. Otherwise fall through to the reference-organism name map (human/mouse/rat/zebrafish/fly/yeast — these are MyGene.info named-string exceptions, not numeric IDs).

```python
# Source: src/ct/tools/_species.py — existing public API
from ct.tools._species import resolve_species_taxon

def _normalize_mygene_species(species: str) -> str:
    s = (species or "").strip().lower()
    if not s:
        return s
    if s.isdigit():
        return s

    # Reference organisms: MyGene.info accepts named strings for these only
    _REFERENCE_ORGANISM_MAP = {
        "human": "human", "homo sapiens": "human",
        "mouse": "mouse", "mus musculus": "mouse",
        "rat": "rat",
        "zebrafish": "zebrafish", "danio rerio": "zebrafish",
        "drosophila": "fly",
        "yeast": "yeast", "saccharomyces cerevisiae": "yeast",
    }
    if s in _REFERENCE_ORGANISM_MAP:
        return _REFERENCE_ORGANISM_MAP[s]

    # Plant species and all others: resolve via YAML registry → numeric taxon ID
    taxon_id = resolve_species_taxon(species)  # passes original casing
    if taxon_id:
        return str(taxon_id)

    # Fallback: try regex extraction then return original
    m = re.search(r"([A-Za-z][a-z]+)\s+([a-z][a-z]+)", s)
    if m:
        candidate = f"{m.group(1)} {m.group(2)}".lower()
        if candidate in _REFERENCE_ORGANISM_MAP:
            return _REFERENCE_ORGANISM_MAP[candidate]
        taxon_id = resolve_species_taxon(candidate)
        if taxon_id:
            return str(taxon_id)

    return species  # last resort: pass through original
```

### Pattern 2: Default Species String Must Match Registry Keys

**What:** The default value for `species` parameters must be a string that `_species.py` can resolve. `_build_lookup()` indexes on `binomial.lower()`, which uses spaces not underscores.

**The bug:** `plant_data.py` line 119 uses `species: str = "arabidopsis_thaliana"` (Ensembl underscore form). The lookup in `_build_lookup()` uses `" ".join(s.lower().split())` so `"arabidopsis_thaliana"` becomes `"arabidopsis_thaliana"` as a key — not found. The registry key is `"arabidopsis thaliana"` (with a space).

**Fixed state:** Change the default to `"Arabidopsis thaliana"` (space-separated binomial). This is consistent with every other tool's default and with the STATE.md decision "Arabidopsis thaliana (3702) as default species across all tools".

```python
# BEFORE (plant_data.py:119)
def load_expression(
    gene: str,
    species: str = "arabidopsis_thaliana",   # WRONG: underscore form
    ...

# AFTER
def load_expression(
    gene: str,
    species: str = "Arabidopsis thaliana",   # CORRECT: matches registry key
    ...
```

### Pattern 3: CLI Branding — Literal String Fix

**What:** Any user-facing instruction that says "run a CLI command" must say `ag` not `ct`.

**The bug:** `downloader.py` line 167 prints `ct config set data.{name} {dest}`. This is the manual-download guidance path (only reached when `auto_download=False`).

**Fixed state:**
```python
# BEFORE (downloader.py:167)
console.print(f"  Then run: [cyan]ct config set data.{name} {dest}[/cyan]")

# AFTER
console.print(f"  Then run: [cyan]ag config set data.{name} {dest}[/cyan]")
```

### Anti-Patterns to Avoid

- **Rebuilding `_MYGENE_SPECIES_MAP` with more entries:** More entries will diverge further from the YAML. Delete the plant species section; keep only the reference-organism exceptions that MyGene requires as named strings.
- **Changing `_species.py` `_build_lookup()` to add underscore keys:** The fix goes in `plant_data.py` (correct the default string). Do not teach `_species.py` to accept underscore forms — that would mask future callers making the same mistake.
- **Using `resolve_species_ensembl_name()` in `parity.py`:** This returns underscore form for Ensembl URLs. MyGene.info wants numeric taxon IDs for plant species, not Ensembl-style names.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Species name → taxon ID | Custom dict in `parity.py` | `resolve_species_taxon()` from `_species.py` | Single source of truth; YAML-backed; already correct for all other tools |
| Species default value for plant tools | Ensembl underscore form | Space-separated binomial `"Arabidopsis thaliana"` | Registry keys are space-separated; underscore form silently fails lookup |

**Key insight:** The infrastructure (`_species.py` + `species_registry.yaml`) already exists and is already correct. The bugs are purely call-site errors — wrong string format and a forgotten reference to the infrastructure.

## Common Pitfalls

### Pitfall 1: lru_cache invalidation after fixing `_species.py` calls

**What goes wrong:** `_build_lookup()` is cached with `@lru_cache(maxsize=1)`. If tests import the module before patching the registry, the cache is warm and tests may see stale results.

**Why it happens:** `lru_cache` persists across test function calls within a single pytest session unless explicitly cleared.

**How to avoid:** The existing `test_species.py` pattern uses `@patch` on `_load_registry` and calls `_build_lookup.cache_clear()` before each test that needs registry isolation. Follow the same pattern in any new tests.

**Warning signs:** Test passes in isolation but fails when run with full suite.

### Pitfall 2: `_MYGENE_SPECIES_MAP` deletion breaks existing test assertions

**What goes wrong:** `tests/test_parity_tools.py:test_mygene_lookup_species_phrase_normalization` asserts that `"Schistosoma mansoni genes"` normalizes to `"6183"`. Schistosoma mansoni is currently in `_MYGENE_SPECIES_MAP` (line 85). If `_MYGENE_SPECIES_MAP` is removed and `_species.py` is used instead, this species must be either: (a) in `species_registry.yaml`, or (b) handled by the regex fallback calling `resolve_species_taxon`.

**Check:** Schistosoma mansoni (taxon 6183) is NOT currently in `species_registry.yaml`. This test would break if the map is simply deleted without a registry entry or fallback.

**How to avoid:** Either add Schistosoma mansoni (and other parasites currently in `_MYGENE_SPECIES_MAP`) to the YAML registry, or keep a slim `_MYGENE_SPECIES_MAP` for non-plant, non-reference species that are not in the registry and not worth adding. The cleanest approach is: keep only the reference-organism named-string exceptions in a small in-code map; rely on `_species.py` for all registry species; let unlisted species fall through to the original value (current behavior for parasites). The parasites are low-priority species — their taxon IDs are already hardcoded correctly and do not need YAML-backing. The key goal is that plant species added to YAML are automatically recognized.

**Warning signs:** `test_mygene_lookup_species_phrase_normalization` fails after the refactor.

### Pitfall 3: Existing `test_plant_data.py` passes `"arabidopsis_thaliana"` as species

**What goes wrong:** Several test functions in `test_plant_data.py` call `load_expression(species="arabidopsis_thaliana", ...)`. After fixing the default, the tests pass the underscore form explicitly. Since `@validate_species` calls `resolve_species_binomial("arabidopsis_thaliana")` → returns `""` (not in registry) → emits a "not in registry" warning → now `"species_warning"` is in the result. This could break test assertions that do NOT expect a warning.

**Check:** `test_load_expression_finds_gene`, `test_load_expression_gene_not_found`, `test_load_expression_tissue_filter`, `test_load_expression_data_not_found`, and `test_load_expression_case_insensitive_gene` all pass `species="arabidopsis_thaliana"` explicitly. They pass an absolute `dataset` path, so the decorator DOES run. After the default-fix, these tests with explicit underscore species will start emitting warnings. Currently they pass because they don't assert on the absence of `species_warning`.

**Resolution:** When fixing the default, also update these test call sites to use `"Arabidopsis thaliana"` (space form). This ensures the tests reflect real agent usage and do not silently test a broken species string.

**Warning signs:** Tests pass but a stray `species_warning` appears in results that weren't expecting one.

### Pitfall 4: The `test_downloader.py` `test_expected_datasets_present` assertion

**What goes wrong:** `test_expected_datasets_present` asserts exactly `{"depmap", "prism", "l1000", "msigdb", "string", "alphafold"}`. The `plantexp` entry was added to `DATASETS` in Phase 2 but this test was written before that addition.

**Check:** Looking at `downloader.py`, `plantexp` IS now in DATASETS (lines 84-98). The test at line 24 of `test_downloader.py` will therefore ALREADY fail. This is a pre-existing test debt, not introduced by Phase 2.1.

**Resolution:** Update `test_expected_datasets_present` to include `"plantexp"` in the expected set. This is a test fix that comes along with Phase 2.1 work.

**Warning signs:** `test_expected_datasets_present` fails — it already does, independently of Phase 2.1 changes.

### How `_build_lookup()` constructs keys (why underscore fails)

```python
# Source: src/ct/tools/_species.py lines 56-73
for entry in _load_registry():
    taxon_id: int = entry["taxon_id"]
    binomial: str = entry["binomial"]
    # Canonical binomial (lowercase) → "arabidopsis thaliana" (SPACE, not underscore)
    lookup[binomial.lower()] = value
    # Common names
    for name in entry.get("common_names", []):
        lookup[" ".join(name.lower().split())] = value
    # Abbreviations
    for abbrev in entry.get("abbreviations", []):
        lookup[abbrev.lower()] = value
```

```python
# src/ct/tools/_species.py line 111
```
