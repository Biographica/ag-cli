# Phase 2.2: Integration Fixes II - Research

**Researched:** 2026-02-26
**Domain:** Python CLI routing (Typer entry point passthrough) and Python package dependency declaration (PyYAML in pyproject.toml)
**Confidence:** HIGH

## Summary

Phase 2.2 is a two-bug gap-closure phase with surgically precise fixes. The v1.0 re-audit found two integration defects that survived Phase 2.1:

**Bug 1 — CLI routing:** The `entry()` function in `src/ct/cli.py` (lines 1450-1481) implements a routing dispatch that intercepts `sys.argv` and prepends `"run"` to arguments that are not in a `passthrough` set. This set (lines 1453-1471) includes `"config"`, `"data"`, `"tool"`, `"trace"`, `"knowledge"`, `"keys"`, `"doctor"`, `"setup"`, `"release-check"`, `"report"`, `"case-study"`, `"bench"`, and `"run"`, but is **missing `"species"`**. As a result, `ag species list` becomes `["run", "species", "list"]` — which routes to the hidden `run` command and treats `"species list"` as a natural-language query string rather than dispatching to the `species_app` subcommand. The fix is one line: add `"species"` to the passthrough set.

**Bug 2 — PyYAML dependency:** `PyYAML` is used in two files — `src/ct/tools/_species.py` (line 40: `import yaml`) and `src/ct/data/manifest.py` (line 40: `import yaml`) — but `pyyaml` is absent from the `[project] dependencies` list in `pyproject.toml`. The package is currently installed in the development environment (PyYAML 6.0.1), which masks the missing declaration. A clean `pip install ag-cli` in a fresh virtualenv would succeed without PyYAML and then fail at runtime when either `ag species list` or `load_manifest()` is called.

Both fixes are fully diagnosed. No architectural decisions are required. No new libraries are introduced. The plan for this phase is a single PLAN covering both fixes plus the tests.

**Primary recommendation:** Add `"species"` to the `passthrough` set in `entry()`, add `"pyyaml>=6.0"` to `[project] dependencies` in `pyproject.toml`, and write two tests (one per fix) — both test files already exist.

## Architecture Patterns

### Pattern 1: Typer entry() passthrough set

**What:** The `entry()` function in `cli.py` is the `[project.scripts]` target (`ag = "ct.cli:entry"`). It intercepts `sys.argv` to route "plain" invocations (`ag "question"`, `ag`) to the hidden `run` subcommand, while passing through named subcommands unchanged. The passthrough set is an explicit allowlist of subcommand names.

**The bug:**
```python
# src/ct/cli.py lines 1452-1481
def entry():
    """Package entry point."""
    argv = list(sys.argv[1:])
    passthrough = {
        "config",
        "data",
        "tool",
        "trace",
        "knowledge",
        "keys",
        "doctor",
        "setup",
        "release-check",
        "report",
        "case-study",
        "bench",
        "run",
        "--help",
        "-h",
        "--install-completion",
        "--show-completion",
        # "species" is MISSING
    }

    if not argv or argv[0] not in passthrough:
        argv = ["run", *argv]  # "species list" → ["run", "species", "list"]

    app(args=argv, prog_name="ag")
```

**The fix:** Add `"species"` to the passthrough set.

```python
    passthrough = {
        "config",
        "data",
        "tool",
        "trace",
        "knowledge",
        "keys",
        "doctor",
        "setup",
        "release-check",
        "report",
        "case-study",
        "bench",
        "run",
        "species",  # ADD THIS
        "--help",
        "-h",
        "--install-completion",
        "--show-completion",
    }
```

**Why this works:** `species_app` is registered with `app.add_typer(species_app, name="species")` at line 297. Typer handles dispatch from there — `entry()` only needs to not corrupt the argv before passing to `app()`.

**Key detail:** The existing test `test_manifest.py::test_species_list_command` tests `runner.invoke(app, ["species", "list"])` — invoking the Typer app directly, bypassing `entry()`. This test passes today but does NOT cover the routing bug. The new test must test `entry()` specifically.

### Pattern 2: PyYAML dependency declaration

**What:** `pyproject.toml` `[project] dependencies` is the canonical list of required runtime packages. Any package used at runtime must be listed here to ensure correct installation in clean environments.

**The bug:**
- `src/ct/tools/_species.py` line 40: `import yaml`
- `src/ct/data/manifest.py` line 40: `import yaml`
- `pyproject.toml` `[project] dependencies`: PyYAML is absent.

**The fix:** Add one line to `pyproject.toml`:

```toml
[project]
dependencies = [
    "typer>=0.12",
    "rich>=13.0",
    "prompt-toolkit>=3.0",
    "anthropic>=0.74.0",
    "openai>=1.0",
    "claude-agent-sdk>=0.1",
    "httpx>=0.27",
    "pandas>=2.0",
    "numpy>=1.24",
    "scipy>=1.10",
    "python-dotenv>=1.0",
    "markdown>=3.5",
    "nbformat>=5.7",
    "pyyaml>=6.0",  # ADD THIS — used by _species.py and manifest.py
]
```

**Version pin rationale:** PyYAML 6.0 is the stable modern release; `safe_load` API has not changed across 6.x. Specifying `>=6.0` ensures a modern release while allowing patch updates. The current installed version is 6.0.1.

**PyYAML package name:** The pip/PyPI name is `pyyaml` (lowercase), the Python import name is `yaml`. This is the correct declaration format for pyproject.toml.

### Pattern 3: Testing entry() routing

**What:** The `entry()` function must be tested by monkeypatching `sys.argv` and patching `ct.cli.app` to capture the `args` argument. This pattern is already established in `tests/test_cli.py`.

**Existing pattern (from `test_cli.py` lines 61-76 and 79-94):**

```python
def test_entry_routes_plain_invocation_to_hidden_run(monkeypatch):
    called = {}

    def fake_app(*, args, prog_name):
        called["args"] = args
        called["prog_name"] = prog_name

    monkeypatch.setattr("ct.cli.app", fake_app)
    monkeypatch.setattr("sys.argv", ["ag", "profile", "FLC"])

    from ct.cli import entry
    entry()

    assert called["args"] == ["run", "profile", "FLC"]


def test_entry_preserves_explicit_subcommand(monkeypatch):
    called = {}

    def fake_app(*, args, prog_name):
        called["args"] = args
        called["prog_name"] = prog_name

    monkeypatch.setattr("ct.cli.app", fake_app)
    monkeypatch.setattr("sys.argv", ["ag", "config", "show"])

    from ct.cli import entry
    entry()

    assert called["args"] == ["config", "show"]
```

**New test for `species` passthrough** follows this exact pattern:

```python
def test_entry_preserves_species_subcommand(monkeypatch):
    called = {}

    def fake_app(*, args, prog_name):
        called["args"] = args
        called["prog_name"] = prog_name

    monkeypatch.setattr("ct.cli.app", fake_app)
    monkeypatch.setattr("sys.argv", ["ag", "species", "list"])

    from ct.cli import entry
    entry()

    assert called["args"] == ["species", "list"]
```

### Anti-Patterns to Avoid

- **Testing species dispatch via `runner.invoke(app, ["species", "list"])`:** This bypasses `entry()`. The test must use the monkeypatch pattern to verify that `entry()` does not prepend `"run"` to the `species` argument. The existing `test_species_list_command` in `test_manifest.py` is a valid test but covers a different code path.
- **Pinning PyYAML to an exact version (`==6.0.1`):** Use `>=6.0` to allow patch updates. Exact pins in `pyproject.toml` are inappropriate for library-style packages.
- **Using `pyyaml` in the `[project.optional-dependencies]` section:** PyYAML is a required runtime dependency (not optional) — it's needed for `ag species list` and manifest loading. It belongs in `[project] dependencies`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| YAML loading | Custom YAML parser | `pyyaml` / `yaml.safe_load()` | Already in use; `safe_load` is the secure YAML API (no arbitrary code execution) |
| CLI subcommand routing | Custom string manipulation in `entry()` | Add to `passthrough` set | The set is the intentional design; add missing entries, don't redesign the function |

**Key insight:** Both bugs are missing-one-line errors. No redesign is needed.

## Common Pitfalls

### Pitfall 1: Test covers Typer app but not entry() routing

**What goes wrong:** `test_manifest.py::test_species_list_command` calls `runner.invoke(app, ["species", "list"])` and passes. This test exercises the Typer subcommand itself but does NOT exercise `entry()`. A developer might see this test passing and conclude routing is fine.

**Why it happens:** `typer.testing.CliRunner.invoke()` calls the Typer app object directly, bypassing the `entry()` wrapper. The routing bug only manifests through the installed script entry point.

**How to avoid:** Write a separate test for `entry()` using the monkeypatch pattern (see Pattern 3 above). The new test must call `entry()` directly with `sys.argv = ["ag", "species", "list"]` and assert `args == ["species", "list"]` (not `["run", "species", "list"]`).

**Warning signs:** Only `test_species_list_command` exists; no test monkeypatches `sys.argv` with `"species"`.

### Pitfall 2: PyYAML already installed masks the declaration gap in tests

**What goes wrong:** All tests pass in the current development environment because PyYAML 6.0.1 is installed. The missing declaration is invisible until a clean install attempt.

**Why it happens:** `pip install -e ".[dev]"` installs declared dependencies. PyYAML was installed previously (possibly as a transitive dependency of another tool) and persists in the environment without being declared.

**How to avoid:** The test for this fix should verify the declaration is present in `pyproject.toml` directly (parse the TOML and assert `"pyyaml"` appears in `dependencies`). This is the only automated check that validates the declaration, not the presence of the installed package.

**Warning signs:** All tests pass but `pyyaml` is absent from `pyproject.toml` `[project] dependencies`.

### Pitfall 3: Wrong passthrough format for `--version` flag

**What goes wrong:** When implementing the fix, there's temptation to wonder whether `--version` should be in passthrough. It should NOT — `--version` is handled by the `run` subcommand's `--version`/`-V` flag (line 1194-1208 in cli.py). The `run` subcommand checks `if version: console.print(...)`. The existing passthrough set already handles `--help`/`-h` correctly.

**Why it happens:** The entry() routing is non-obvious because `--version` does not appear in passthrough but still works correctly.

**How to avoid:** Only add `"species"` to passthrough — do not modify any other entries or add `"--version"`.

### Pitfall 4: Confusing PyPI package name with import name

**What goes wrong:** In `pyproject.toml`, the package is declared as `pyyaml` (PyPI/pip name), but in Python code it is imported as `yaml` (`import yaml`). Developers sometimes declare it as `yaml` in pyproject.toml, which resolves to a different (unrelated) package on PyPI.

**How to avoid:** Use `"pyyaml>=6.0"` in `pyproject.toml` — lowercase `pyyaml` is the correct PyPI package name.

## Open Questions

1. **Should the pyproject.toml test use `tomllib` (stdlib, Python 3.11+) or `tomli` (backport)?**
   - What we know: The project requires `>=3.10`. `tomllib` is stdlib only from 3.11. For Python 3.10, `tomllib` is absent; `tomli` is the backport.
   - What's unclear: Whether the CI/dev environment targets Python 3.10 or 3.11+.
   - Recommendation: The test can avoid the TOML parsing issue entirely by reading the raw file text and asserting `"pyyaml"` appears as a substring. This is simpler than adding a test-only dependency on `tomli` and equally precise for this purpose.

2. **Are there other subcommands missing from the passthrough set?**
   - What we know: Phase 2.2 only needs to fix `"species"`. The re-audit identified this specific gap.
   - What's unclear: Whether future subcommands will be forgotten in the same way.
   - Recommendation: Fix only `"species"` for this phase. Consider adding a comment to the passthrough set noting that new `app.add_typer()` additions must be mirrored here — this is a low-cost guard against the pattern recurring.
